Tasks vs Work Queue Items

Choosing the right runtime model for your module

Advantage
Limitation
Criteria
Tasks (Threads)
Work Queue Items
Memory Usage
High (Own stack required per task)
Low (Shared stack on queue)
Blocking IO / Sleep
Allowed (px4_usleep, poll)
Not Allowed (Must be non-blocking)
Scheduling Model
OS Scheduler (Preemptive)
Cooperative / Event-driven
CPU Isolation
Good (Isolated thread priority)
Risk (Can starve peers on same queue)
Debug Visibility
Shows in `top` command
Use `work_queue status` (grouped)
Ideal Use Case
Long computations, blocking operations, drivers with slow I/O
High-rate periodic tasks, event-driven logic, drivers with fast/interrupt I/O
Pro Tip: Default to Work Queue Items for new modules unless you specifically need blocking behavior.
